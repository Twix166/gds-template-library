# Purpose

Record functional, non-functional, policy and operational requirements in a structured way.

# When to use this

Use from Discovery onwards when requirements need to be visible, reviewed and traced.

# Inputs

- User needs
- Policy intent
- Research findings
- Technical and operational constraints

# Template

Add requirement ID, description, type, source, priority, owner, status, acceptance evidence and links.

# Quality checklist

- Requirements are clear and testable where possible.
- Each requirement has a source.
- Requirements are linked to user needs or service outcomes.
