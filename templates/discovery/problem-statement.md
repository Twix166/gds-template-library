# Purpose

Define the problem the team is exploring, the people affected and the outcomes the service should support.

# When to use this

Use during Discovery and revisit whenever the understanding of the problem changes.

# Inputs

- User research findings
- Policy or operational context
- Existing service data
- Stakeholder input

# Quality checklist

- The problem is framed from user and service outcomes, not a preferred solution.
- Evidence and assumptions are clearly separated.
- Scope and exclusions are explicit.
