# Purpose

Define the test approach, responsibilities, environments and evidence needed for Beta.

# When to use this

Use when moving from prototype learning to build, integration and release readiness.

# Inputs

- Requirements catalogue
- Non-functional requirements
- Architecture overview
- Release plan

# Quality checklist

- Testing covers user journeys and service risks.
- Non-functional requirements have test evidence.
- Defects and acceptance decisions are traceable.

# Field glossary

Hover over table headers, and some field-name cells, to see short definitions in renderers that support HTML title text.

- **Acceptance decision owner**: The person accountable for accepting test evidence or release readiness.
- **Decision**: The decision made or needed.
- **Definition**: A precise explanation of what the measure or severity means.
- **Environments**: The environments covered by testing or release activity.
- **Evidence produced**: The test output, report or record generated.
- **Exit criteria**: The conditions that must be met before testing is considered complete.
- **Field**: The name of the information item to complete.
- **In scope test types**: The kinds of testing included in the strategy.
- **Out of scope**: What is deliberately excluded.
- **Owner**: The person or role accountable for the item.
- **Purpose**: Why the activity, test or artefact exists.
- **Release impact**: How the issue severity affects release decisions.
- **Requirement or risk**: The requirement or risk being tested or traced.
- **Response**: The content to complete for the field.
- **Response expectation**: The expected response time or action for the severity level.
- **Service scope covered**: The part of the service covered by the strategy.
- **Severity**: The seriousness of an issue or incident.
- **Status**: The current state of the item.
- **Test evidence**: Evidence generated through testing.
- **Test type**: A category of testing, such as accessibility, security or performance testing.
