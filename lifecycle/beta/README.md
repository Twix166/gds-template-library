# Beta

Beta builds and tests the service with real users. Use Beta to prove the service can meet user needs, operate reliably, protect users and support a move to Live.

## Recommended artefacts

| Stage of Beta | Artefact | What it is used for | How to use it | Referenced in GDS guidance |
| --- | --- | --- | --- | --- |
| Start | [Beta delivery plan](../../templates/beta/beta-delivery-plan.md) | Plan delivery, releases, dependencies and governance. | Keep scope tied to evidenced user needs and the MVP. | [Beta phase](https://www.gov.uk/service-manual/agile-delivery/how-the-beta-phase-works); [Service Standard point 7](https://www.gov.uk/service-manual/service-standard/point-7-use-agile-ways-of-working). |
| Release planning | [Release plan](../../templates/beta/release-plan.md) | Plan releases, communications, monitoring and rollback. | Use for private beta, public beta and major release decisions. | [Service Standard point 14](https://www.gov.uk/service-manual/service-standard/point-14-operate-a-reliable-service); [Deploying software regularly](https://www.gov.uk/service-manual/technology/deploying-software-regularly). |
| Quality assurance | [Test strategy](../../templates/beta/test-strategy.md) | Define test scope, approach, responsibilities and evidence. | Link tests to user journeys, requirements and service risks. | [Service Standard point 14](https://www.gov.uk/service-manual/service-standard/point-14-operate-a-reliable-service); [Service Standard point 8](https://www.gov.uk/service-manual/service-standard/point-8-iterate-and-improve-frequently). |

## Suggested sequence

1. Confirm Beta scope, delivery plan and release approach.
2. Build around user journeys and service risks.
3. Run research, testing and accessibility work continuously.
4. Keep privacy, security and operations evidence current.
5. Use readiness checks before private beta, public beta and Live transition.
6. Prepare the evidence pack from linked artefacts, not duplicated material.
