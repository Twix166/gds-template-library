# Discovery Evidence Flow

This page shows how Discovery artefacts build on each other. Teams do not need to produce every artefact separately if a smaller set of documents tells the story clearly.

## Flow

```text
Initial brief or request
  -> problem statement
  -> research and stakeholder planning
  -> user needs catalogue
  -> journey map and service blueprint
  -> assumptions log
  -> options assessment
  -> discovery findings report
  -> discovery assessment evidence pack
  -> decision: stop, re-scope, proceed to Alpha, or hand over
```

## Artefact relationships

| Artefact | Uses evidence from | Feeds into | Key question it answers |
| --- | --- | --- | --- |
| Problem statement | Initial brief, stakeholder input, early research | User research plan, assumptions log, options assessment | What problem are we investigating? |
| Stakeholder map | Team knowledge, governance routes, operational context | Research access, decision log, evidence pack | Who needs to be involved? |
| User needs catalogue | User research, analytics, support data, existing insight | Journey map, findings report, options assessment | What do users need, and how do we know? |
| Current state journey map | Research observations, interviews, analytics, support data | Service blueprint, findings report, Alpha hypotheses | What happens to users today? |
| Service blueprint current state | Journey map, operational interviews, system and policy analysis | Options assessment, architecture discovery, dependency register | How does the current service work behind the scenes? |
| Assumptions log | Problem statement, research gaps, team decisions | Research planning, options assessment, Alpha plan | What are we assuming, and how risky is it? |
| Options assessment | Findings, constraints, assumptions, stakeholder input | Findings report, phase decision | What should we do next? |
| Discovery findings report | All Discovery artefacts | Assessment evidence pack, Alpha planning, governance decision | What did we learn and recommend? |
| Discovery assessment evidence pack | Findings report and evidence artefacts | Service assessment or internal review | How does the evidence map to the Service Standard? |

## Evidence maturity

Use confidence levels consistently:

| Confidence | Meaning | Typical evidence |
| --- | --- | --- |
| Low | Plausible but weakly evidenced. Needs validation before major decisions. | Stakeholder view, one or two observations, untested assumption. |
| Medium | Supported by several sources but still incomplete. Good enough for Discovery decisions if risk is understood. | Multiple interviews, support data, analytics, operational walkthroughs. |
| High | Strong evidence from repeated patterns or reliable data. | Consistent research findings, robust service data, validated operational evidence. |

## Decision points

| Decision | Evidence needed |
| --- | --- |
| Stop | Evidence that the problem is not worth solving, cannot be solved cost-effectively, is already solved elsewhere, or is better addressed through policy, content or operational change. |
| Re-scope | Evidence that the original brief is too broad, too narrow, solution-led or focused on the wrong part of the service. |
| Proceed to Alpha | Evidence that there is a real user problem, likely value in solving it, viable ideas to test and manageable constraints. |
| Hand over | Evidence that another team or organisation owns the most useful next step. |

## Using the evidence pack

The evidence pack should not duplicate every document. It should link to the best available evidence and explain what each item proves.

For example:

| Service Standard area | Strong evidence | Common gap |
| --- | --- | --- |
| Understand users and their needs | User needs catalogue linked to research notes and current journey evidence. | Needs are asserted but not linked to evidence. |
| Solve a whole problem for users | Problem statement, journey map and blueprint showing the wider problem. | Discovery focuses only on one transaction or system. |
| Provide a joined-up experience | Blueprint and stakeholder map showing teams, channels and dependencies. | Offline, support or operational channels are missing. |
| Make sure everyone can use the service | Research plan and findings include disabled users, support needs and digital exclusion. | Accessibility is deferred until Beta. |
| Define success measures | Findings report and options assessment include candidate measures and baselines. | Success is described as delivery completion rather than user or service outcomes. |
